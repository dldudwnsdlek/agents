# 최종 지원 결정 (apply-decide)

## 언제 쓰나 (triggers)
- 채용 파이프라인의 **5단계(최종)**로 실행될 때
- 여러 공고 중 어디에 지원할지 우선순위를 정해야 할 때

## 입력
- `workspace/02_evaluated.json` — 적합도 점수
- `workspace/03_analysis.md` — 기업 매력도/리스크
- `workspace/04_cover_letters/*.md` — 자소서 강도
- `PORTFOLIO.md` — 지원자 선호(선택)

## 의사결정 프레임
회사별로 다음을 종합해 `apply` / `hold` / `drop` 부여:
- **적합도**(평가자 `fit_score`) — 합격 가능성의 토대
- **회사 매력도/리스크**(분석가 별점·리스크) — 가서 좋을 곳인가
- **자소서 강도**(설득력 있는 스토리가 나왔는가)
- **투입 대비 효과(ROI)** — 마감 임박도·지원 난이도·지원자 선호
→ 종합 점수로 **추천 순위**(1순위, 2순위 …)를 부여.

## 절차 (steps)
1. 위 입력들을 모두 읽어 회사별로 4개 축을 정리한다.
2. 각 회사에 `apply`/`hold`/`drop`과 **한 줄 사유 + 다음 액션**을 부여한다.
   - `hold`는 "무엇이 충족되면 apply로 바꿀지" 조건을 명시.
3. `apply` 회사를 추천 순위로 정렬한다(마감 임박·합격 기댓값 우선).
4. 결과를 `workspace/05_decision.md`에 저장하고, 맨 위에 **오늘 할 일 Top 3**를 제시한다.

## 출력 형식 — `workspace/05_decision.md`
```markdown
# 최종 지원 결정 (YYYY-MM-DD)

## ✅ 오늘 할 일 Top 3
1. [회사] — 마감 D-3, 즉시 지원 (자소서: 04_cover_letters/회사.md)
2. ...
3. ...

## 결정 표
| 순위 | 회사 | 직무 | 결정 | 적합도 | 회사별점 | 자소서 | 마감 | 한 줄 사유 |
|---|---|---|---|---|---|---|---|---|
| 1 | ... | ... | apply | 82 | ★★★★ | 강 | D-3 | ... |
| - | ... | ... | hold | 68 | ★★★ | 중 | 상시 | (조건: …) |
| - | ... | ... | drop | 61 | ★★ | - | - | (사유: …) |

## 코멘트
- 보류(hold) 항목별 전환 조건 …
- 앞 단계 재작업 권고(있으면) …
```

## 흔한 함정 (pitfalls)
- 새 공고 수집·재평가·자소서 재작성을 직접 하지 않는다. 필요하면 앞 단계로 돌려보내라고 **권고**만.
- 모든 공고를 "일단 지원"으로 뭉뚱그리지 않는다 — 핵심은 **걸러내기와 집중**.
- 사유 없는 결정 금지. 모든 결정에 근거를 댄다.

## 성공 기준 (verify)
- `05_decision.md`에 모든 후보 회사가 apply/hold/drop으로 분류됨.
- 각 결정에 사유와 다음 액션이 있다.
- "오늘 할 일 Top 3"와 우선순위 표가 존재한다.
